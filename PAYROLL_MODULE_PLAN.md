# 工资管理模块分阶段需求分析与开发计划

## 1. 文档目的

本文档用于规划“小公司本地账务管理系统”的工资管理模块。

工资发放不建议只作为普通财务支出处理。普通支出只能表达“钱出去了”，但无法清楚管理：

1. 每个员工应发多少。
2. 每个员工实发多少。
3. 哪些工资已经发放。
4. 哪些工资还未发放。
5. 哪些工资已经调整、作废或留痕。
6. 工资发放后对应哪笔财务流水。

因此，建议新增独立的“工资管理”模块，并让它和现有财务流水、账户余额、月度支出打通。

## 2. 已确认的第一版边界

根据当前确认，第一版工资模块按以下边界开发：

1. 菜单名称使用“工资管理”。
2. 工资第一版不关联项目。
3. 工资作为独立财务信息管理。
4. 发放工资时生成公司工资支出流水。
5. 工资支出不影响项目利润。
6. 工资批次允许同一个月份创建多个批次，因为可能存在周结工资。
7. 工资明细字段包括：
   - 基本工资
   - 全勤
   - 话补
   - 奖金
   - 提成
   - 扣款
8. 预留但不参与自动计算的字段：
   - 社保
   - 公积金
   - 个税
9. 已发放工资后续应允许修改和作废，但必须留痕。
10. 第一版先建立操作记录和状态规则；复杂联动作废流水可后续增强。

## 3. 总体设计原则

工资模块继续遵守当前系统架构：

```text
React 页面层
↓
renderer/api 前端 API 调用层
↓
preload 安全桥接
↓
Electron IPC 层
↓
Service 业务逻辑层
↓
Repository 数据访问层
↓
SQLite 数据库
```

设计原则：

1. React 页面不直接操作 SQLite。
2. 数据库访问放在 repository 层。
3. 工资计算、状态流转、发放规则放在 service 层。
4. 工资金额全部使用整数分。
5. 工资不得物理删除，只能软删除或作废。
6. 作废工资不得参与工资统计。
7. 工资发放最终必须生成财务流水，不能绕开现有账户和报表体系。
8. 第一版不做复杂税务、社保、公积金自动计算。
9. 第一版不做多用户审批流。
10. 第一版不修改项目利润口径。

## 4. 业务对象

### 4.1 工资批次

工资批次表示一次工资核算或发放。

批次可以按月，也可以按周。

示例：

```text
2026年7月第一周工资
2026年7月工资
```

工资批次负责表达：

1. 工资月份。
2. 批次名称。
3. 发放日期。
4. 发放账户。
5. 批次状态。
6. 应发合计。
7. 扣款合计。
8. 实发合计。
9. 关联财务流水。

### 4.2 工资明细

工资明细是批次下的员工工资行。

每个员工一行，记录：

1. 员工。
2. 基本工资。
3. 全勤。
4. 话补。
5. 奖金。
6. 提成。
7. 扣款。
8. 社保。
9. 公积金。
10. 个税。
11. 应发工资。
12. 实发工资。
13. 备注。

第一版计算规则：

```text
应发 = 基本工资 + 全勤 + 话补 + 奖金 + 提成
实发 = 应发 - 扣款
```

说明：

1. 社保、公积金、个税字段第一版只记录，不参与自动计算。
2. 后续如果要支持多个扣款项，可以新增工资扣款明细表。

## 5. 状态流转

工资批次状态建议：

```text
draft       草稿
confirmed   已确认
paid        已发放
voided      已作废
```

规则：

1. 草稿可以编辑。
2. 草稿可以软删除。
3. 已确认可以发放。
4. 已确认可以取消确认。
5. 已发放后应保留修改、作废留痕。
6. 已作废工资不参与工资统计。
7. 已发放工资如修改，应记录调整原因。
8. 已发放工资如作废，应记录作废原因。

## 6. 工资和财务流水的关系

第一版工资不关联项目。

发放工资时，一个工资批次生成一条公司工资支出流水。

生成流水字段建议：

```text
direction = expense
fundType = salary
category = 人工工资
accountId = 工资发放账户
projectId = 空
employeeId = 空
affectsReceivable = false
affectsProjectProfit = false
status = normal
```

这样现有功能可以自然统计：

1. 账户余额减少。
2. 月度支出增加。
3. 工资支出进入公司支出口径。
4. 项目利润不受工资模块影响。

## 7. 第一版建议数据表

### 7.1 payroll_batches

工资批次表。

```text
id
month
name
pay_date
account_id
status
total_gross_cents
total_deduction_cents
total_net_cents
paid_transaction_id
voided_at
void_reason
remark
created_at
updated_at
deleted_at
```

说明：

1. `month` 格式为 `YYYY-MM`。
2. `pay_date` 格式为 `YYYY-MM-DD`。
3. `account_id` 是发放工资账户。
4. `paid_transaction_id` 记录发放后生成的财务流水。
5. 合计字段由 service 根据明细计算。

### 7.2 payroll_items

工资明细表。

```text
id
batch_id
employee_id
base_salary_cents
attendance_bonus_cents
phone_allowance_cents
bonus_cents
commission_cents
deduction_cents
social_insurance_cents
housing_fund_cents
tax_cents
gross_salary_cents
net_salary_cents
remark
created_at
updated_at
deleted_at
```

### 7.3 payroll_operation_logs

工资操作留痕表。

```text
id
batch_id
item_id
action
detail
created_at
```

建议 action：

```text
create
update
delete
confirm
cancel_confirm
pay
void
adjust
```

说明：

1. 第一版用于记录工资关键操作。
2. 后续可以扩展为更完整的审计日志。

## 8. 阶段十六：工资管理基础版

### 阶段目标

完成工资批次、工资明细、工资确认、工资发放和财务流水生成的最小闭环。

### 功能范围

1. 新增“工资管理”菜单。
2. 新增工资批次列表。
3. 支持按月份筛选工资批次。
4. 支持新增工资批次。
5. 支持编辑工资批次。
6. 支持软删除草稿工资批次。
7. 支持查看工资批次详情。
8. 支持在工资批次下新增员工工资明细。
9. 支持编辑工资明细。
10. 自动计算应发工资和实发工资。
11. 自动汇总批次合计。
12. 支持确认工资。
13. 支持取消确认。
14. 支持发放工资。
15. 发放工资时生成一条公司工资支出财务流水。
16. 发放成功后批次状态变为已发放。
17. 已发放工资允许作废，但必须填写原因。
18. 已发放工资允许调整，但必须记录操作。
19. 可以查看工资批次关联的财务流水。

### 暂不实现

2. 社保、公积金自动计算。
3. 工资条导出。
4. 银行代发文件。
5. 多次部分发放。
6. 发放后自动反向冲销。
7. 审批流。
8. 考勤管理。
9. 多扣款项明细。

## 13. 风险与边界

### 13.1 税务风险

工资薪金可能涉及个人所得税、社保、公积金等政策。

第一版不自动计算税费，避免错误政策口径导致使用风险。

### 13.2 财务口径风险

工资明细本身不直接进入财务报表。

只有“发放工资”后生成的财务流水，才进入账户余额和月度支出。

第一版工资不进入项目利润。

### 13.3 作废风险

工资发放后已经生成财务流水。

第一版允许工资批次作废和调整，并记录操作日志。

是否联动作废财务流水，放到阶段十六 B 具体实现时再确定。

### 13.4 数据复杂度风险

不要第一版就做完整人事薪酬系统。

当前系统定位是小公司本地账务管理，因此第一版工资模块只做“工资核算 + 发放入账 + 留痕”。
